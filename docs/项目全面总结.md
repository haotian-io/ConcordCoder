## 2. 研究背景与核心痛点：三重认知鸿沟

主流 AI 编程助手多依赖「描述一句 → 直接出码」的 **Vibe Coding**；RAG 或「写错再改」的事后循环在复杂仓库上常陷入**同向反复试错**。可解释性不足可概括为 **Triple Cognitive Gap**：

1. **Model–Intent Gap**：自然语言与隐含边界未对齐，模型对意图理解偏题。  
2. **Memory–History Gap**：开发者对历史设计约束、团队约定记忆不全。  
3. **Surface–Implicit Gap**：隐式 API/数据契约未显式进入模型上下文。

**ConcordCoder** 将 **认知对齐** 前置于大段生成：先显式化上下文、约束与不确定性，再 **约束驱动生成**；与事后修补相对，主叙事是 **事前对齐（pre-hoc）**。论文中对应 **ContextBundle → AlignmentRecord → 约束生成与认知摘要** 流水线，并以 **Constraint-Grounded Alignment Dialogue** 与 **Confidence-Guided Probing** 为两条可写主线。

**目标投稿**：ASEJ 特刊 *Explainability in Automated Software Engineering*；`Paper/main.tex` 为 Springer Nature 系文类，截稿等以稿内与期刊为准。

## 3. 三阶段架构：叙事与代码落点

### 3.1 叙事层（在做什么）

- **Phase 1 上下文抽取**  
  不立刻生成，先做「体检」：**AST**、**调用图**、**Git 历史**、**测试线索**，聚合成可审计片段与 `RiskItem` 等。  

- **Phase 2 认知对齐**  
  在 Phase 1 证据上发起**多轮/多阶段**对话（可 LLM 可规则），将假设约束沉淀为可溯源的 **AlignmentRecord**（与「闲聊」区分的是**状态与约束**随轮次更新）。  

- **Phase 3 约束生成与探针**  
  在硬约束下生成；**Confidence-Guided Probing** 用 token 级置信度（有 API 时 logprobs）与文件 churn 等，对高不确定、高风险区域**生成探针问题**（**设计**上还包含对齐全局后的 **Cognitive Summary** 与**约束违规时的再生成**；**当前实现**中整段重试与产品化弹窗/span 级重写程度见 `Paper/main.tex` Limitations 与 `constrained_gen` 实现）。

### 3.2 模块与类型（与仓库路径对应，均在 `Code/src/concordcoder/` 下）

| 阶段 | 作用 | 主要模块 / 类型 |
|------|------|-----------------|
| **Phase 1** | 静态+Git+测试 → 片段/风险/调用图 | `extraction/bundle_builder.py`，`ast_analyzer.py`，`call_graph.py`，`git_historian.py`，`test_extractor.py`；`ContextBundle`（`schemas.py`） |
| **Phase 2** | 对齐对话 | `alignment/llm_dialogue.py`，`alignment/dialogue.py`；`AlignmentRecord` |
| **Phase 3** | 生成 + 可选锚点/探针 | `generation/constrained_gen.py`；`generation/anchor_pipeline.py`；`generation/probing.py`（`ProbingEngine`） |
| **编排** | 全流程/单任务 | `pipeline.py` |
| **CLI** | 入口 | `cli.py`：`extract` / `align` / `run` / `once` |

**单任务锚点/探针（与 InlineCoder 思想、论文 artifact 一致）**  

- `SingleTaskSpec` 带 `target_file`、`target_symbol`，`use_anchor=True`：调用图 + `draft_anchor` + `assemble_inlinecoder_mvp` → `GenerationRequest.assembly`。  
- `with_probe=True`：在锚点文本上跑 `ProbingEngine`（无 key 时 mock logprobs）→ `SingleTaskResult.probe`。  
- CLI：`concord once` 的 `--target-file`，`--symbol`，`--use-anchor`，`--with-probe`（示例见 `Code/README*.md`）。

## 4. 已交付与进度

1. **论文**  
   `Paper/main.tex` + `ref.bib`，Springer 文类可编译；摘要与 Evaluation 已与 **TaskLab + `mini_eval`** 的 **artifact 机评** 口径对齐，大规模 SWE-bench/CoderEval 等在文中为 **计划研究**。Overleaf 主文档应为 **`main.tex`**（见 `Paper/OVERLEAF.md`；不存在 `0_ACM_main.tex`）。  

2. **代码**  
   `BundleBuilder`、多阶段对齐、`ConstrainedGenerator`、锚点管线、`ProbingEngine`、`run_single_task` 与 `concord once` 等已贯通。**`concord run` / `once` / `align` 与 `scripts/mini_eval.py` 必须配置 `OPENAI_API_KEY` 或 `ANTHROPIC_API_KEY`（可选 `OPENAI_BASE_URL`）**；缺省则**报错退出**。**仅** Phase 1 的 `concord extract` 可在无 key 下做纯静态分析。自动化测试用 `tests/conftest.StubLLM` 模拟 LLM，不访问公网。

## 5. 测试与 LLM 要求

- **`concord run` / `once` / `align` 与 `scripts/mini_eval.py` 必须配置 `OPENAI_API_KEY`（或 Anthropic）**；可选 **`OPENAI_BASE_URL`** 指向 OpenAI 兼容中转。缺 key 或 API 异常时**直接报错退出**，不再生成无 Key 桩。  
- **Phase 1 `extract`** 仍可在无 key 下只做静态抽取。  
- `pytest` 使用 `tests/conftest.py` 中的 **`StubLLM`** 注入，不访问公网。  
- 机评与论文表述一致时，不将 fixture 上 pytest 的数值**外推**为「工业大库已证」；大规模指标见文中有计划的 RQ1 段。
