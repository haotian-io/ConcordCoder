# ConcordCoder：你要做什么 — 详细指南

本文面向**你自己或合作者**：说明项目目标、当前已有内容、接下来按阶段该完成什么、如何验收与写论文时怎么对应模块。

---

## 1. 项目在解决什么问题

**现象**：代码生成往往直接给出补丁，用户难以判断模型为何这样改、是否符合意图、是否触碰不该动的约束；用户隔一段时间接手旧项目时，自己的记忆也会衰减。

**主张**：在生成代码之前，先做两件事——（1）从仓库里抽取与本次任务相关的证据与约束线索；（2）通过面向人的对话，让用户与系统达成对「事实 + 可改动边界」的**共同理解**。然后再做约束下的生成与验证。

**论文级表述**：把流程拆成 **上下文抽取与融合 → 认知对齐对话 → 约束驱动的代码生成（含验证闭环）**。

---

## 2. 当前仓库里已经有什么（起点）

目录：`ConcordCoder/`（Python 包，使用 `src/concordcoder/` 布局）。

| 模块 | 路径 | 现状 | 你的下一步 |
|------|------|------|------------|
| 数据契约 | `schemas.py` | 定义了 `ContextBundle`、`AlignmentRecord`、`Constraint` 等 | 随实验需要扩展字段（例如引用 commit、测试名） |
| 上下文抽取 | `extraction/bundle_builder.py` | **基线**：关键词命中 + 代码窗口片段 | 替换/增强为：向量检索、调用图裁剪、AST 符号索引 |
| 对齐对话 | `alignment/dialogue.py` | **基线**：固定问题清单 + 无 LLM 的草案 | 接入真实多轮对话（LLM）、把回答写回 `AlignmentRecord` |
| 生成 | `generation/stub.py` | **桩**：输出 Markdown 式「计划」 | 接入真实 patch 生成 + `pytest`/lint 失败反馈迭代 |
| 流水线 | `pipeline.py` | 串联 extract → align → stub | 与评测脚本、用户研究流程对齐 |
| CLI | `cli.py` | `concord extract`、`concord run` | 增加子命令：`align`（交互）、`eval`（跑基准子集）等 |

**环境**：在项目根目录使用虚拟环境（已有 `.venv` 时可 `source .venv/bin/activate`），安装：`pip install -e ".[dev]"`。

**运行**：

```bash
concord extract <仓库路径> --task "自然语言任务"
concord run <仓库路径> --task "自然语言任务"    # 生成 CONCORD_PLAN.md（桩）
```

## 3. 总体目标拆解（你要交付的「东西」）

### 3.1 工程上

1. **可复现流水线**：给定仓库路径 + 任务描述 +（可选）用户回答，产出可追溯的产物：`context_bundle.json`、`alignment_record.json`、补丁或计划、以及评测日志。
2. **两条可选增强路径**（至少深耕一条写到论文里）：
   - **检索 + 静态分析**：比纯关键词更准的上下文包；
   - **验证闭环**：生成 → 跑测试/类型检查 → 将错误反馈给模型再改。

### 3.2 研究上（对应你的 RQ）

| 研究问题 | 你需要准备什么 |
|----------|----------------|
| **RQ1 质量**：不同方法生成的结果是否更好 | 自动化任务集（如 SWE-bench 子集）、指标：resolved、测试通过、补丁规模等 |
| **RQ2 主客观**：用户是否更理解系统、更信任修改 | 用户实验：主观量表 + **客观小题**（例如「哪条约束被违反」「哪类文件不应改」） |
| **RQ3 成本收益**：对齐是否值得 | 记录轮次、时间、token、任务完成步数；比较「直接生成」vs「先对齐」 |

## 4. 分阶段任务清单（建议顺序）

### 阶段 A：把「数据契约」钉死（约 1 周）

**你要做**：

- [ ] 确定 `ContextBundle` 最小可用字段：任务摘要、证据片段（路径+行号）、**证据等级**（测试/类型/实现/注释）、**开放问题**列表。
- [ ] 确定 `AlignmentRecord`：**Hard / Soft 约束**如何区分；是否与 CI 脚本一一对应。
- [ ] 约定磁盘产物命名与版本（例如 `artifacts/<run_id>/`），便于实验复现。

**验收**：任意一次运行能落盘 JSON，且第二轮用同一份 JSON 可复现「生成桩」的输出（确定性或记录 seed）。

---

### 阶段 B：上下文抽取升级（约 2–4 周）

**你要做**：

- [ ] **检索**：在 `BundleBuilder` 同类接口下增加 embedding 检索（代码块/文件级）；保留关键词基线作消融对照。
- [ ] **静态分析（择一深入）**：Python 可用 `libcst`/`ast` + import 图；JS/TS 可用 `typescript` compiler API 或轻量解析；输出「与任务相关的子图」节点列表。
- [ ] **噪声控制**：文档与代码冲突时，在 bundle 里标注冲突与证据等级（对齐你之前的「冲突消解」设想）。

**验收**：在同一批任务上，对比「仅关键词」与「检索+静态」：top-k 片段里相关文件的命中率（你可自建小标注集，例如 20 个任务人工标相关文件）。

---

### 阶段 C：对齐对话从「脚本」变成「可用流程」（约 2–4 周）

**你要做**：

- [ ] 设计 **3 段式对话**：意图边界 → 约束确认 → 风险共读（可固定为问卷 + 可选自由补充）。
- [ ] 接入 LLM：输入 = `ContextBundle` + 当前轮次回答；输出 = 更新版 `AlignmentRecord` + 下一轮问题（仅问高不确定性项时可做「最短对齐」消融）。
- [ ] 可选：CLI `concord align` 做终端交互；或做一个最简 Web/VS Code 扩展（后置，非必须）。

**验收**：盲评或小规模试点（3–5 人）能走完流程，且最终 `AlignmentRecord` 可被下游「硬约束校验」消费（见阶段 D）。

---

### 阶段 D：约束驱动的生成与验证闭环（约 3–6 周）

**你要做**：

- [ ] 将 `AlignmentRecord` **编译**为可执行检查：例如「禁止改动的路径 glob」「必须通过的测试列表」「公开 API diff 检测」。
- [ ] 生成：调用模型产出 unified diff 或补丁；应用补丁；运行 `pytest`/`npm test`/…；失败则将日志压缩后反馈给模型（多轮上限 N）。
- [ ] 记录：每轮耗时、token、失败原因类别（类型错误 / 测试失败 / 约束违反）。

**验收**：在子集 SWE-bench 或自建任务上，**Full 方法**（对齐+约束闭环）对比 **Baseline**（直接生成）有可报告的提升或至少可分析的失败模式。

---

### 阶段 E：用户研究 / 正式实验（与阶段 D 可部分并行）

**你要做**：

- [ ] **试测**：同一批任务做 within-subject 或 between-subject 设计，平衡学习效应（任务顺序轮换）。
- [ ] **RQ2 客观题**：每题绑定「唯一正确选项」，来自 `ContextBundle`/`AlignmentRecord` 中的事实与约束。
- [ ] **伦理与数据**：知情说明、录屏与数据保留策略、不收集敏感密钥。

**验收**：预注册或实验方案文档里写清假设、样本量估算、统计方法（即便是探索性分析也要写清）。

---

### 阶段 F：论文与材料（持续）

**你要做**：

- [ ] **系统图**：抽取 / 对齐 / 生成三线 + 产物 JSON。
- [ ] **消融表**：无检索、无对齐、无验证闭环、全流程。
- [ ] **威胁**：任务难度偏差、SWE-bench 数据泄漏讨论、对齐对话过长导致疲劳等。

## 6. 与你论文结构的一一映射（写稿时可直接当目录）

1. **Introduction**：问题（黑箱生成 + 记忆衰减）→ 主张（对齐再生成）。
2. **Related Work**：代码大模型、仓库级 agent、spec/约束生成、人机对齐与可解释修复；你列的 arXiv / IEEE 条目需逐篇写清**差异点**（你强调的是**维护场景下的认知对齐与可审计约束**，不是单纯多轮 chat）。
3. **Problem & Formulation**：定义 `ContextBundle`、`AlignmentRecord`、什么是「违背硬约束」。
4. **Method**：三阶段 + 可选实现细节（检索、静态分析、校验闭环）。
5. **Experimental Setup**：数据集、基线、指标、用户研究协议。
6. **Results & Discussion**：RQ1–RQ3 + 消融 + 威胁。
7. **Conclusion & Future Work**：成本、可扩展语言、更强的形式化约束。

## 7. 风险与对策（提前写进实验设计）

| 风险 | 对策 |
|------|------|
| 基准任务太难，全员挂 | 分层 easy/medium；报告分层结果 |
| 对齐流程太长 | 「最短对齐」模式，只确认 top 不确定性 |
| 静态分析误报污染约束 | 证据分级 + 默认不进 hard，除非用户确认 |
| 评审质疑「只是多轮 prompt」 | 突出**结构化产物**与**硬约束可验证性**，并用消融展示对话结构的价值 |
