## 失败根因分析

### 1. gemini-3.1-pro-preview

**现象**: diff_len=142，但无法提取文件路径

**根因**: 输出格式非标准。Gemini生成的diff缺少 `--- a/` 和 `+++ b/` 头部行，直接使用 `+` 和 `-` 表示修改，但没有完整的unified diff文件头。

**解决方案**: 
- 在prompt中增加严格的格式示例
- 增强解析器以识别简化diff格式

### 2. deepseek-v4-pro

**现象**: diff_len=0，返回空内容 (174s超时)

**根因**: API端点对该模型的响应异常，可能是端点不支持此模型名称或超时设置过短。

**解决方案**: 
- 验证API端点是否支持deepseek-v4-pro
- 增加超时时间到300秒以上
- 尝试使用deepseek-chat替代

### 3. MiniMax-M2.7

**现象**: diff_len=2103，但FHR=0

**根因**: diff格式正确但包含多行修改，文件路径解析成功但无法正确匹配目标文件。可能是diff中修改了多个文件而非目标文件。

**解决方案**: 
- 检查diff内容是否包含目标文件separable.py
- 增加路径匹配的回退策略

### 4. kimi-k2.6

**现象**: diff_len=117，但FHR=0

**根因**: diff内容过短(117字节)，可能只做了微小的修改或未包含目标文件。

**解决方案**: 
- 检查diff具体内容
- 增加格式约束提示

### 5. glm-5.1

**现象**: diff_len=363，但FHR=0

**根因**: diff存在但文件路径不匹配目标文件。

**解决方案**: 
- 检查diff中的文件路径
- 增加路径规范化处理

## 改进建议

1. **模型适配层**: 为不同模型添加输出格式适配器，处理非标准 diff 格式
2. **格式强制**: 在 prompt 中增加更严格的格式示例和约束
3. **模型名称验证**: 测试 API 端点实际支持的模型名称列表
4. **回退策略**: 当 diff 解析失败时，尝试从代码块或自然语言中提取修改内容
5. **约束增强**: 对齐阶段增加格式相关的显式约束（如 "输出必须是标准 unified diff 格式"）
6. **并发测试修复**: 使用LLMClient的api_key/base_url参数而非环境变量，避免线程竞争

## 结论

失败主要原因是 **输出格式不兼容** 和 **API端点支持问题**。3/8模型成功（gpt-5.5, qwen3.5-plus, deepseek-v4-flash），其余5个模型需要格式适配或API配置调整。
